# Студенческий планер

## Описание приложения
«Студенческий планер» — это учебное Android-приложение, разработанное в рамках лабораторной работы для изучения навигации в Jetpack Compose. Приложение позволяет студентам просматривать список дисциплин текущего семестра, получать подробную информацию по каждому предмету, а также просматривать свой профиль и изменять настройки.

## Реализованные экраны
- **Главный экран (Home)**: Отображает список всех учебных дисциплин в виде карточек. Каждая карточка содержит название предмета, имя преподавателя, количество баллов и текущую оценку.
- **Экран деталей (Details)**: Показывает расширенную информацию о выбранной дисциплине, включая полное описание курса.
- **Экран профиля (Profile)**: Содержит информацию о студенте: ФИО, группа, факультет, email и статистику успеваемости.
- **Экран настроек (Settings)**: Позволяет изменять настройки приложения, такие как включение push-уведомлений и темная тема.

## Используемые технологии
- **Kotlin**: Основной язык программирования.
- **Jetpack Compose**: Современный набор инструментов для построения пользовательского интерфейса.
- **Navigation Compose**: Библиотека для реализации навигации между экранами с поддержкой передачи параметров и управления стеком.

## Схема навигации
Навигация в приложении построена с использованием `NavHost` и `NavController`. Все маршруты централизованно описаны в `sealed class Screen`, что обеспечивает типобезопасность.

Схема переходов между экранами:

- Главный экран является стартовым (`startDestination`).
- При клике на дисциплину на главном экране выполняется переход на экран деталей с передачей ID предмета в качестве параметра.
- Кнопки «Профиль» и «Настройки» на панели приложения осуществляют переход на соответствующие экраны.
- Возврат на предыдущий экран осуществляется с помощью системной кнопки «Назад» или стрелки «Назад» в верхней панели.

## Ответы на контрольные вопросы

1. Что такое NavController и для чего он используется?
   NavController — это центральный компонент, управляющий навигацией в приложении. Его основная роль заключается в:

Выполнении переходов: Он предоставляет метод navigate(), который позволяет перемещаться между экранами (destinations).

Управлении стеком (Back Stack): Он хранит историю посещённых экранов и управляет ей. С помощью метода popBackStack() можно вернуться на предыдущий экран.

Обработке параметров: Он отвечает за передачу данных между экранами.

Создавать его через rememberNavController() важно, потому что remember гарантирует, что экземпляр NavController будет сохранён между рекомпозициями (перерисовками UI). Если создать его заново при каждом обновлении интерфейса, навигация сломается, а состояние стека будет потеряно.

2. Как передать параметр в маршрут навигации?
   Процесс передачи параметра состоит из трёх этапов:

Определение маршрута с плейсхолдером: В sealed class (или при регистрации в NavHost) маршрут описывается с фигурными скобками, например "details/{subjectId}".

Переход с параметром: При вызове navController.navigate("details/123") строка "123" подставляется на место плейсхолдера.

Извлечение параметра: В NavHost при регистрации экрана используется backStackEntry.arguments?.getString("subjectId"), чтобы получить значение параметра и передать его в @Composable функцию.

Разница между обязательными и опциональными параметрами:

Обязательные являются частью пути (path) и всегда должны присутствовать (например, "details/{id}"). Если их не передать, навигация не сработает.

Опциональные передаются через query-параметры (например, "search?query={query}"). Для них можно задать значение по умолчанию (defaultValue), и если параметр не передан, он не вызовет ошибки.

3. Зачем использовать sealed class для маршрутов?
   Использование sealed class (например, sealed class Screen(val route: String)) даёт несколько преимуществ по сравнению с обычными строками:

Типобезопасность (Type Safety): Компилятор знает обо всех возможных экранах. Если вы попытаетесь перейти по маршруту, которого нет в sealed class, код не скомпилируется.

Централизация: Все маршруты хранятся в одном месте. Это упрощает поддержку кода — если нужно изменить название маршрута, достаточно сделать это в одном файле, а не искать строки по всему проекту.

Вспомогательные функции: Можно добавить функции для построения маршрутов (как createRoute в примере), что предотвращает ошибки конкатенации строк.

Пример ошибки, которую sealed class помогает избежать:
Без sealed class легко ошибиться в написании строки: navController.navigate("detals/123") вместо "details/123". С sealed class вы будете вынуждены использовать существующий объект: navController.navigate(Screen.Details.createRoute("123")), что гарантирует корректность маршрута.

4. Что такое Back Stack и как им управлять?
   Back Stack (стек назад) — это структура данных, работающая по принципу LIFO (Last In, First Out). Она хранит историю переходов пользователя. Каждый раз, когда пользователь переходит на новый экран, он добавляется на вершину стека. При нажатии кнопки «Назад» текущий экран удаляется из стека, и пользователь возвращается на предыдущий.

Схема back stack для последовательности Home → Profile → Settings:

Начальное состояние: [Home] (вершина стека — Home)

Переход на Profile: [Home, Profile]

Переход на Settings: [Home, Profile, Settings] (Settings — на вершине)

Что произойдёт при вызове popBackStack() на экране Settings?
Будет удалён верхний элемент (Settings), и стек станет [Home, Profile]. Пользователь вернётся на экран Profile. Если вызвать popBackStack() на Home (когда в стеке только он), приложение закроется, так как больше нет экранов для отображения.

5. Как работает startDestination в NavHost?
   startDestination — это обязательный параметр NavHost, который указывает, какой экран должен быть показан первым при запуске приложения. Он задаётся строкой, соответствующей одному из зарегистрированных маршрутов.

Можно ли изменить startDestination динамически?
Да, но это требует ручного управления состоянием. NavHost сам по себе не поддерживает смену стартового экрана после создания. Однако можно реализовать логику, при которой в зависимости от условий (например, авторизован ли пользователь) вы будете программно вызывать navController.navigate() после запуска, или использовать условную логику внутри NavHost, чтобы показывать разные графы навигации.

6. Что произойдёт, если навигировать на несуществующий маршрут?
   Если вызвать navController.navigate("non_existent_route"), где такого маршрута нет в NavHost, приложение не вылетит (не произойдёт краша), но и ничего не произойдёт. NavController просто проигнорирует этот вызов, так как он не может найти подходящий destination.

Как можно обработать такую ситуацию?
Чтобы избежать подобных проблем, рекомендуется:

Использовать sealed class для маршрутов, чтобы компилятор проверял их существование на этапе написания кода.

Настроить логирование в NavHost, чтобы в процессе отладки видеть, какие маршруты вызываются.

При необходимости, можно обернуть вызов navigate в проверку, но обычно это излишне при грамотном проектировании.

7. Зачем нужен параметр launchSingleTop в навигации?
   Параметр launchSingleTop (или launchSingleTop = true в navigate()) предотвращает создание дублирующихся экземпляров одного и того же экрана на вершине стека.

Пример проблемы без него:
Представьте экран деталей товара, на котором есть кнопка «Перейти к похожему товару», которая ведёт на тот же самый экран деталей, но с другим ID. Если пользователь нажмёт на неё несколько раз, в стеке может оказаться [Details(id=1), Details(id=2), Details(id=3)]. Чтобы вернуться к первому товару, пользователю придётся нажать «Назад» три раза.

Как он влияет на back stack:
С launchSingleTop = true, если вы пытаетесь перейти на экран, который уже находится на вершине стека, новый экземпляр не создаётся. Вместо этого существующий экземпляр получает новые аргументы (обновляется), а стек не разрастается. Это делает навигацию более удобной для пользователя и предотвращает бесконтрольное разрастание back stack.
